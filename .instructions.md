# dtuGateway Development Workflow Instructions

## Prepare for Commit Workflow

### Purpose
When you have uncommitted changes ready to commit, use this workflow to automatically:
1. Update the CHANGELOG.md with user-facing changes
2. Stage modified files
3. Generate a suggested commit message
4. **(You will manually review and commit)**

### Usage

Say any of these commands to trigger the workflow:
- `"prepare for commit with changelog: [Description of changes]"`
- `"prepare commit: [Description]"`
- `"prep commit changelog: [Description]"`

### Example

**User**: "prepare for commit with changelog: Enhanced DTU stability with improved connection handling"

**System will**:
1. Review current uncommitted changes
2. Update CHANGELOG.md - Add entry under [Unreleased] with user's description
3. Stage all modified files
4. Generate and display a commit message template
5. Display git diff summary

### Workflow Details

#### Changelog Updates
- Automatically adds user's description to the appropriate category (Changed/Added/Fixed based on description keywords)
- Adds current date timestamp to entry
- Maintains chronological order

#### File Staging
- Stages all modified files listed by `git status`
- Includes CHANGELOG.md update
- Files ready for `git commit`

#### Commit Message Format
```
[Category] Concise summary of changes

- Detailed point 1
- Detailed point 2

Changelog: [User description from prepare for commit]
```

#### Manual Next Steps
After the workflow output:
1. Review the staged changes: `git diff --staged`
2. Review the suggested commit message
3. Adjust if needed: `git reset` and edit files manually
4. Commit when ready: `git commit`

### Important Notes

- ✅ The agent will **NOT** commit automatically
- ✅ You have full control over commit message and content
- ✅ Files are staged but not committed
- ✅ Review everything before committing

### Reverting Staged Changes

If you need to unstage files before committing:
```bash
git reset
```

To discard uncommitted changes entirely:
```bash
git checkout -- .
```

## Changelog Guidelines

### User-Focused Entries
Write entries that users care about, not implementation details.

**✅ Good**:
- "Improved DTU connection stability"
- "Enhanced WiFi signal handling"
- "Added support for Waveshare 1.28-inch displays"

**❌ Avoid**:
- "Refactored dtuInterface.cpp"
- "Fixed variable naming in webserver.cpp"
- "Updated include/dtuInterface.h"

### Category Selection
- **Added**: New features, new display types, new API endpoints
- **Changed**: Improvements to existing functionality, performance optimizations
- **Fixed**: Bug fixes, documentation corrections
- **Removed**: Deprecated or removed features
- **Security**: Security-related updates

### Date Format
Changelog entries use ISO date format: `YYYY-MM-DD`

## Development Practices

### Branch Strategy
- `main` - Stable releases
- `develop` - Development branch for next release

### Before Pushing
1. Run tests/builds locally
2. Verify CHANGELOG.md is up-to-date
3. Review commit messages for clarity

### PR Guidelines
- Link related issues (#110, #113, etc.)
- Include user-facing description in PR body
- Reference changelog updates
